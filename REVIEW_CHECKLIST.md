# Чек-лист: тестирование, код-ревью и security review перед коммитом

Инструкция описывает порядок действий, который был применён при подготовке
коммита `abba3eb` (исправление флоу вопросов и онбординга в `bot.py`), —
чтобы повторять его для будущих изменений в этом проекте.

## Когда применять

Перед коммитом любого нетривиального изменения в `bot.py`, `index.html`
или другие рабочие файлы бота-копирайтера. Не нужно для правок опечаток,
текстов сообщений бота или изображений.

---

## Шаг 1 — Тесты

1. Если тестов на затронутую функцию ещё нет — написать их (`tests/`):
   `pytest` + `unittest.mock` для `bot.py`, `Playwright` для `index.html`.
2. Запустить весь набор:
   ```bash
   pytest tests/ -v
   ```
3. Если нашли баг во время написания тестов — сначала зафиксировать его
   тестом, который **падает** (это доказывает, что баг реален и тест его
   действительно ловит), только потом чинить код и добиваться, чтобы тест
   стал зелёным.

## Шаг 2 — Код-ревью

1. Можно вызвать встроенный скилл `/code-review` (или `/code-review high`
   для более широкого покрытия) — он проверит текущий diff на баги,
   дублирование и упрощения.
2. Либо вручную: перечитать diff и проверить каждую ветку `if`/`try` на
   реальные крайние случаи — пустые данные, ошибки внешнего API (Groq),
   рассинхронизацию состояния в `user_data`.
3. Найденные баги, которые реально бьют по пользователям в обычном
   сценарии, — чинить сразу. Остальное — в Roadmap (см. Шаг 4).

## Шаг 3 — Security review

Только чтение, никаких изменений в этом шаге.

1. Можно вызвать встроенный скилл `/security-review` для проверки
   pending-изменений.
2. Для полного аудита проекта (не только diff):
   - **Секреты в коде** — grep по трекаемым файлам на паттерны
     `api[_-]?key|token|secret|password` и т.п.:
     ```bash
     git ls-files | xargs grep -nEi '(api[_-]?key|token|secret|password)\s*[:=]\s*["\x27][A-Za-z0-9_-]{10,}'
     ```
   - **`.gitignore`** — убедиться, что `.env` в нём есть (и в
     `nejrosveta-agent/.gitignore`, если меняется тот проект).
   - **История git** — `.env` не должен был коммититься ни разу, включая
     «висящие» (unreachable) объекты, оставшиеся от `git add` без коммита:
     ```bash
     git rev-list --objects --all | grep -i "\.env"
     git fsck --unreachable --no-reflog
     ```
   - **Зависимости** — известные CVE в `requirements.txt`:
     ```bash
     pip install pip-audit
     pip-audit -r requirements.txt
     ```
3. Уязвимость с реальной эксплуатируемостью в нашем коде — чинить сразу.
   Уязвимость в неиспользуемом коде-пути библиотеки — в Roadmap.

## Шаг 4 — Документация

Обновить (устаревшая документация хуже её отсутствия — правило из
`CLAUDE.md`):

- **`README.md`** → раздел «Тестирование, код-ревью и security review» —
  что прогнали, что нашли и исправили.
- **`README.md`** → раздел «Roadmap» — что нашли, но осознанно отложили,
  и почему (риск, приоритет, что мешает исправить сразу).
- **`ARCHITECTURE.md`** — если поменялась логика или структура кода.
- **`.env.example`** — если появилась новая переменная окружения.

## Шаг 5 — Коммит и пуш

1. `git status` — проверить, что в staging не попадает посторонний мусор
   (на Desktop рядом с репозиторием может лежать много несвязанных
   файлов — добавлять только то, что реально относится к изменению).
2. Коммит-сообщение: что сделали + что именно исправили, без воды.
3. `git push origin main` — только после того, как пользователь явно
   попросил отправить на GitHub.

---

## Как решить: чинить сразу или в Roadmap

| Чинить сразу | В Roadmap |
|---|---|
| Баг воспроизводится в обычном сценарии использования бота | Баг воспроизводится только при нестандартном/маловероятном вызове |
| Уязвимость эксплуатируема через код, который реально выполняется | Уязвимый код-путь библиотеки в проекте не используется |
| Исправление укладывается в объём текущей задачи | Исправление требует отдельной задачи большего масштаба |
