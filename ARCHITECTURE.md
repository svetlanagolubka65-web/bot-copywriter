# Архитектура: ИИ-помощник по контенту (bot.py)

## Стек

| Компонент | Технология |
|-----------|-----------|
| Язык | Python 3.10+ |
| Telegram | python-telegram-bot 21.5 (async) |
| ИИ-модель | Groq API → Llama 3.3 70B Versatile |
| Хранилище пользователей | `users.json` + `users_data.json` (локальные файлы) |
| Конфиг | `.env` через python-dotenv |

## Как работает

```
Пользователь (Telegram)
        │
        ▼
  start() ──── новый юзер? ──▶ gender_keyboard() ──▶ handle_callback(gender_*) ──▶ send_onboarding()
        │
        ▼
  handle_message()
  handle_callback()
        │
        ▼
  ask_next_question()  ←── CONTENT_TYPES[тип]["questions"]
        │
        ▼ (после сбора всех ответов)
  generate_from_answers()
        │
        ▼
  Groq API (Llama 3.3 70B)
        │
        ▼
  Сообщение пользователю + result_keyboard()
```

## Ключевые структуры

### Доступ (owner-only)
`check_owner()` сверяет `update.effective_user.id` со списком `OWNER_IDS` (из `ADMIN_IDS` в `.env`, через запятую если ID несколько). Вызывается первой строкой в `start`, `settov_command`, `help_command`, `handle_message`, `handle_callback` — если ID не совпал, пользователь получает «бот приватный» и обработка прерывается. Если `ADMIN_IDS` не задан, ограничение не действует (для совместимости при локальном тестировании).

### CONTENT_TYPES
Словарь с 5 типами контента. Каждый тип содержит:
- `questions` — список уточняющих вопросов (текст + опционально кнопки)
- `prompt` — шаблон промпта с `{плейсхолдерами}`
- `label` — название для логов

### Поток сообщений

1. Пользователь нажимает кнопку меню → `content_type` сохраняется в `user_data`
2. Бот задаёт вопросы по очереди через `ask_next_question()`
3. Ответ принимается и через кнопку (`qans_*`), и обычным текстом — даже если у вопроса есть кнопки (см. `handle_message`, ветка «Ответ на вопрос в потоке»)
4. Ответы сохраняются в `user_data["answers"]` по ключу вопроса
5. После последнего вопроса → `generate_from_answers()`
6. Промпт собирается через `prompt.format(**answers)`
7. Если пользователь загрузил стиль через `/settov` — он добавляется в системный промпт

### Онбординг и профиль
1. Новый пользователь → `start()` сохраняет его в `users.json` и показывает `gender_keyboard()`
2. Выбор пола (`gender_f` / `gender_m`) → `handle_callback` сохраняет профиль в `users_data.json` (`save_user_profile`) и вызывает `send_onboarding(query.message.chat, name)`
3. Пол хранится в `user_data["gender"]` и влияет на `_gender_note()` — от какого лица генерируется текст

### Callback-кнопки после генерации

| callback_data | Действие |
|--------------|----------|
| `regenerate` | Повторная генерация с теми же ответами |
| `refine` | Запрашивает правки, перегенерирует с учётом |
| `hashtags` | Подбирает 7–10 хэштегов к последнему тексту |
| `main_menu` | Очищает `user_data`, кроме `name`/`gender`/`tov`, возвращает в меню |

### Стиль автора (ToV)
- Команда `/settov` → `user_data["waiting_tov"] = True`
- Следующий текст от пользователя сохраняется в `user_data["tov"]`
- При генерации `tov` добавляется в системный промпт: `"Стиль автора:\n{tov}"`

## Хранение данных

- **`users.json`** — список Telegram ID пользователей (для различения новых и вернувшихся)
- **`users_data.json`** — профиль пользователя: имя и пол
- **`user_data`** — in-memory контекст разговора (сбрасывается при /start или main_menu)

## Ограничения

- `user_data` не персистентный — после перезапуска бота стиль (ToV) сбрасывается
- `users.json` / `users_data.json` хранятся локально — при деплое нужно либо монтировать volume, либо перейти на БД
- `generate_from_answers()` падает с необработанным `TypeError`, если `answers` содержит `topic`, но не хватает остальных полей формата (см. Roadmap в `README.md`) — в штатном флоу вопросов бота не воспроизводится

## Тестирование

Полное покрытие описанной выше логики — в [`tests/`](tests/) (pytest + Playwright для лендинга). Список проверенных сценариев и статус код-ревью/security review — в `README.md` → «Тестирование, код-ревью и security review».

# Архитектура: бот-консультант по услугам (intake_bot.py)

Отдельный Telegram-бот, не связанный с `bot.py`. Открывается по deep-link с плашек услуг на лендинге (`index.html`, раздел «Обо мне») и отвечает клиенту вместо прямого перехода в личку Светланы.

## Стек

Тот же, что у `bot.py`: `python-telegram-bot` (async), Groq API (`llama-3.3-70b-versatile`), `.env` через `python-dotenv`. Свой токен — `INTAKE_BOT_TOKEN`. Данные пользователей нигде не хранятся (все состояние — в `user_data`, сбрасывается с ботом).

## Как работает

```
Плашка на лендинге (?start=<slug>)
        │
        ▼
  start() ── slug известен? ──▶ _show_service_card(chat, slug)
        │                              │
        │ (нет / неизвестен)           ├─▶ [❓ Задать вопрос]  ──▶ mode="qa"
        ▼                              └─▶ [✍️ Оставить заявку] ──▶ mode="brief"
  _show_services_list()
        │
        ▼
  handle_message() / handle_callback()
        │
   ┌────┴────┐
   ▼         ▼
 mode=qa   mode=brief
   │         │
   ▼         ▼
Groq +    ask_next_brief_question() ×3
KNOWLEDGE_BASE   │
   │             ▼
   ▼         _finish_brief()
ответ клиенту    │
(+ уведомление    ▼
 ADMIN_IDS,    _notify_admins() + подтверждение клиенту
 если не уверен)
```

## Ключевые структуры

- **`SERVICES`** — 7 услуг (slug → title/desc/includes/price), содержимое продублировано из `.comp-item` в `index.html`. **При изменении цен/состава услуг на лендинге нужно вручную обновить и здесь** — единого источника данных нет.
- **`KNOWLEDGE_BASE`** — `SERVICES` + `GENERAL_FAQ` (те же 6 вопросов, что в `.faq-item` на лендинге), собирается в системный промпт `QA_SYSTEM_PROMPT` для режима свободных вопросов. Инструкция модели — отвечать только по этим фактам, не выдумывать цифры.
- **`BRIEF_QUESTIONS`** — 3 фиксированных вопроса для брифа (задача / сроки / бюджет), общие для всех услуг, кроме `visual` (см. ниже).
- **`PHOTO_STYLES`** — 5 пресетных стилей AI-фотосессии (slug → title + готовый промпт для Gemini), используются только в специальном флоу услуги `visual`. Тексты промптов дублируются в [`prompts/photo_styles.md`](prompts/photo_styles.md) для правки без кода — при изменении держать оба места в синхроне.
- **`CUSTOM_PROMPT_TEMPLATE`** — шаблон промпта для варианта «✍️ Своё описание»: описание клиента подставляется в общий каркас (ракурс/техфлаги/сохранение идентичности), см. тот же `prompts/photo_styles.md`.

## Три режима диалога

### `mode = "qa"` (свободный вопрос)
Текст пользователя уходит в Groq с `QA_SYSTEM_PROMPT`. Если Groq упал (`except Exception`) или ответ содержит один из `NO_ANSWER_MARKERS` («уточню у светланы» и т.п.) — вопрос дополнительно пересылается всем `NOTIFY_CHAT_IDS` через `context.bot.send_message`.

### `mode = "brief"` (сбор заявки)
Повторяет паттерн `bot.py` (`ask_next_question`/`save_answer_and_continue`), но без кнопочных вопросов — только текст. После 3-го ответа `_finish_brief()` сразу шлёт бриф всем `NOTIFY_CHAT_IDS` и отвечает клиенту подтверждением — без промежуточного шага «подтвердите отправку». Используется для всех услуг, **кроме `visual`**.

### Услуга `visual` — сначала подтип, потом свой флоу

Клик «✍️ Оставить заявку» на карточке услуги `visual` не идёт сразу в `mode="brief"` — сначала бот спрашивает подтип (`_ask_visual_type` → `mode="visual_type"`, кнопки `vtype_photo` / `vtype_card` / `vtype_reel`):

- **📸 Фото** (`vtype_photo`) → запускает мини-флоу заказа AI-фото ниже.
- **💌 Открытка** / **🎬 Ролик** (`vtype_card` / `vtype_reel`) → обычный `mode="brief"` (3 текстовых вопроса, как у остальных услуг), но с `context.user_data["visual_subtype"]` = «Открытка»/«Ролик» — эта строка добавляется отдельной строкой «Тип: …» в уведомление `_finish_brief()`.

### Заказ AI-фото (подтип «Фото») — свой мини-флоу вместо брифа

Вместо трёх текстовых вопросов нужно собрать стиль и фото — отдельная цепочка состояний:

```
vtype_photo ──▶ mode="photo_style"  (photo_style_keyboard: 5 пресетов + «✍️ Своё описание»)
                        │
        style_<key> ────┼──── style_custom
        (пресет)        │      (свой стиль)
        │               │           │
        ▼               │           ▼
 photo_style = title     │   mode="photo_custom_wait" ──▶ (текст) ──▶ photo_style = текст
        │                                                             │
        └───────────────────────────┬─────────────────────────────────┘
                                     ▼
                          mode="photo_upload", photo_count=0
                          (photo_done_keyboard: «✅ Готово, отправить»)
                                     │
                     каждое фото ──▶ handle_photo(): пересылает file_id всем
                     (filters.PHOTO)  NOTIFY_CHAT_IDS через context.bot.send_photo
                                      (подпись со стилем — только на первом фото),
                                      photo_count += 1
                                     │
                          «✅ Готово, отправить» (action="photo_done")
                                     │
                          photo_count == 0? ──▶ просит прислать хотя бы одно фото
                                     │
                                     ▼ (иначе)
                          _finish_photo_order():
                            1) шлёт NOTIFY_CHAT_IDS текст готового промпта
                               (photo_prompt) обычным send_message —
                               БЕЗ parse_mode, т.к. свободный текст клиента
                               (custom-стиль) может сломать разбор Markdown
                            2) подтверждение клиенту («Светлана подготовит
                               фото и пришлёт лично»), mode = None
```

Генерация — полностью ручная: фото клиента и готовый промпт (из `PHOTO_STYLES`/`CUSTOM_PROMPT_TEMPLATE`, см. [`prompts/photo_styles.md`](prompts/photo_styles.md)) приходят в чат уведомлений, Светлана вставляет промпт в Gemini и присылает результат клиенту лично в этот чат. Оплата пока не встроена.

## Куда идут уведомления — `NOTIFY_CHAT_IDS`

По умолчанию — все ID из `ADMIN_IDS` (личные сообщения от бота). Если задать `LEADS_CHAT_ID` (ID группы, например `-5540565018`) — уведомления идут туда вместо `ADMIN_IDS`, что удобнее: можно отвечать на конкретную заявку прямо в группе (Reply), не смешивая с личными тестовыми сообщениями. Узнать ID любого чата — добавить туда бота и написать `/chatid`.

## Единый интерфейс отправки сообщений

В отличие от `bot.py` (где вперемешку используются `message.reply_text()` и `chat.send_message()`), все вспомогательные функции здесь принимают именно **chat**, а не message, и шлют через `chat.send_message(...)` — так один и тот же код работает и из `start()` (где есть только `update.effective_chat`), и из `handle_callback()` (`query.message.chat`), и из `handle_message()` (`update.message.chat`).

## Ограничения

- `SERVICES`/`GENERAL_FAQ` синхронизируются с лендингом вручную — нет общего источника данных с `index.html`.
- Бот создан через @BotFather — [@Superkosultantbot](https://t.me/Superkosultantbot), токен в `.env` (`INTAKE_BOT_TOKEN`), ссылки на 7 плашек в `index.html` уже на него ведут.
- Второй воркер (`worker_intake` в `Procfile`) требует отдельного запуска на хостинге — при деплое с одним процессом (например, Heroku free/Railway с одним сервисом) может не подняться вместе с `worker`; нужен либо отдельный сервис на хостинге под этот воркер, либо ручная проверка, что оба процесса запущены.
- Как и в `bot.py`, `user_data` не персистентный — при перезапуске бота незавершённый диалог (вопрос/бриф в процессе, выбор стиля фото, недосланные фото) теряется.
- Заказ AI-фото не хранится нигде за пределами `user_data` и переписки в `NOTIFY_CHAT_IDS` — если чат уведомлений не задан (`LEADS_CHAT_ID` пустой) и `ADMIN_IDS` пуст, фото и заказ нигде не сохранятся.

## Тестирование

`tests/test_6_intake_bot.py` — карточка услуги по slug, свободные вопросы (уверенный ответ / ошибка Groq / неуверенный ответ → уведомление админам), полный флоу брифа, fallback на неизвестный slug, выбор подтипа услуги `visual` (фото/открытка/ролик) и подстановка `visual_subtype` в уведомление, а также флоу заказа AI-фото (выбор пресета/своего стиля, приём и подсчёт фото, пересылка `NOTIFY_CHAT_IDS`, завершение по кнопке). Конвенции те же, что в тестах `bot.py` (`tests/conftest.py`).

# Архитектура: лендинг (`index.html`)

Статический одностраничник без сборки и бэкенда — открывается напрямую как файл или хостится на GitHub Pages ([svetlanagolubka65-web.github.io/bot-copywriter](https://svetlanagolubka65-web.github.io/bot-copywriter/)). Автодеплой настроен на `git push origin main` — Pages подхватывает изменения `index.html` сам, без ручных шагов.

## Позиционирование (менялось в рамках одной сессии — держать в курсе при новых правках)

Хиро-блок и общая подача страницы прошли через несколько редакций:
1. Изначально — консалтинговый питч («Освойте нейросети и экономьте 3–5 часов в неделю»).
2. Затем — питч на основе портфолио («Мои работы: по вашей истории, а не по шаблону»).
3. **Текущая версия** — позиционирование под аудиторию 50+ («Искусственный интеллект после 50 — легко, интересно и с душой»), с двумя параллельными путями конверсии (кнопки `.hero-cta-card` — научиться / заказать подарок).

Если в следующий раз попросят снова поменять позиционирование — проверять, что во всех местах (хиро, шапка, блок «Обо мне», отзывы) осталась согласованная формулировка, а не смесь старой и новой.

## Порядок секций на странице (сверху вниз)

```
NAV (шапка) → HERO → STORY → PORTFOLIO → REVIEWS → ABOUT (услуги)
  → mid-cta (кнопки) → WHY50 → TONIGHT
  → PROCESS → PRICING → FAQ → QUOTE (цитата) → CAPTURE → CONTACTS → footer
```

Секции обозначены HTML-комментариями (`<!-- HERO -->` и т.д.) — при реордере секций проще всего резать файл по этим маркерам (см. «Как переставляли секции» ниже), а не руками копировать большие блоки через Edit.

- **NAV** — сетка из 3 колонок (`grid-template-columns: auto 1fr auto`): логотип (картинка + «НейроСвета») / **`.nav-tagline`** («Голубева Светлана · консультации и услуги по нейросетям», по центру, скрывается до 900px) / ссылки меню.
- **HERO** — фото + `.hero-cta-grid`: две равнозначные карточки-кнопки (`.hero-cta-card` и `.hero-cta-card.is-gift`), а не один основной CTA.
- **STORY** (`#story`) — короткий личный блок-обращение («Здравствуйте! Меня зовут Светлана...»), без карусели.
- **PORTFOLIO** (`#portfolio`) — см. отдельный раздел ниже.
- **REVIEWS** — `.reviews-grid`: карточки `.review-card.is-photo` (реальный скриншот + подпись) вперемешку с обычной `.review-card` (звёзды + текст + `.review-author`) для отзыва без скриншота (история Веры о победе в конкурсе).
- **ABOUT** (`#about`) — биография + список услуг (`.comp-list`, `<details>`-карточки с иконками-персонажами `.comp-icon-frame`) + подблок `.teacher-story` со своей маленькой каруселью школьных фото (см. «Карусели» ниже — это **второй, независимый** механизм карусели, не тот же, что в портфолио).
- **WHY50** / **TONIGHT** — два одинаково устроенных чек-листа (`.why50-list` / `.tonight-list`, 2 колонки, ✔ / ✅ через `::before`).
- **PROCESS, PRICING, FAQ, QUOTE, CAPTURE, CONTACTS** — не менялись в эту сессию, порядок и содержимое прежние.

## Портфолио (`#portfolio`) — карусели вместо лайтбокса

**Лайтбокс убран полностью** (HTML/CSS/JS, ~150 строк) — раньше клик по мелкой карточке-иконке открывал модалку на весь экран. Теперь работы показаны сразу в двухколоночной сетке (`.wk-grid` → `.wk-col` → `.wk-work`), без клика:

- `.wk-carousel` (`data-wk-car`) — картиночная карусель: `‹`/`›` (`.wk-c-prev`/`.wk-c-next`) + точки (`.wk-c-dots`), JS показывает/скрывает `<img class="on">` по индексу. Обычный `<img>`, не lightbox.
- `.wk-audio-pad` — один или несколько `<audio controls>` подряд (например, карточка «Персональные песни» — 3 плеера в одной карточке).
- `.wk-video` + `.wk-tabs` — видео с кнопками-вкладками (`wkSwapVideo(btn, src)` меняет `src` тега `<video id="wk-video-main">` и подсвечивает активную вкладку).

Медиафайлы лежат в `assets/portfolio/` (реальные файлы, не base64) — извлечены из черновика отдельной html-страницы портфолио, которая передавалась с картинками/аудио/видео как inline `data:` URI (файл весил ~7 МБ). Соглашение об именах: `pf-<тема>.<ext>` (например `pf-card-zhanna.jpg`, `pf-song-detstvo.mp3`, `pf-video-roza.mp4`). Старые работы (`portfolio1.png`, `portfolio2.png`, `portfolio3.png`, `gallery1-9.jpg`, `portfolio-video.mp4`, `portfolio-song.mp3`) остались в корне репозитория как были.

## Карусели — два независимых механизма, не путать

На странице два **разных** набора карусельных стилей/скриптов, которые выглядят похоже, но не связаны:
1. `data-wk-car` / `.wk-c-*` — карусели портфолио (может быть много слайдов, до 9 в карточке «Посты, карусели»).
2. `data-story-car` / `.story-*` — единственная карусель школьных фото в блоке `.teacher-story` внутри «Обо мне».

Если в будущем понадобится карусель где-то ещё третий раз — стоит вынести общую функцию вместо копирования третьего почти-идентичного скрипта.

## Как переставляли секции (для будущих похожих правок)

Файл содержит одну гигантскую строку (~90 КБ) — логотип шапки вшит как base64 `data:image/png;...` прямо в HTML (см. «Ограничения»). Из-за неё `Read` с диапазоном строк, задевающим область шапки, падает по лимиту токенов. При реордере/крупных правках вокруг шапки:
- Не читать эту область через `Read` — использовать `Grep`/`sed -n 'N,Mp' index.html | cut -c1-200`, чтобы обрезать длинные строки для просмотра.
- Для перестановки целых секций — резать файл в Python по маркерам-комментариям (`content.find("<!-- HERO -->")` и т.д.), а не пытаться скопировать большой блок вручную через `Edit`.

## Ограничения

- **Логотип в шапке — inline base64 (~68 КБ бинарных данных, ~90 КБ строка в HTML)**, а не отдельный файл — единственное место на странице, где картинка не вынесена в файл. Раздувает `index.html` и физически неудобно для правок рядом (см. выше). Стоит вынести в `assets/` отдельным файлом, когда дойдут руки — риска в этом нет, просто неаккуратно.
- Позиционирование менялось несколько раз за сессию — тексты услуг (`#about`) и отзывов **не** обновлялись под аудиторию 50+, только структура и хиро/новые блоки. Если будет полная переработка на 50+ — тексты карточек услуг всё ещё «универсальные», не под эту аудиторию.
- Нет общего источника данных между `SERVICES` в `intake_bot.py` и карточками услуг в `index.html` (см. «Ограничения» бота-консультанта выше) — актуально и для лендинга.
